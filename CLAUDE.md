# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

**GMTK2026** is a Unity game-jam project (GMTK Jam) built with **Unity 6 (6000.2.12f1)**, the **Universal Render Pipeline (URP 17.2)** and the new **Input System**. There is one gameplay scene: `Assets/Scenes/SampleScene.unity`.

### Where the code lives

Almost all C# lives under **`Assets/__IP ARCHITECTURE/`** — a reusable, project-agnostic gameplay framework carried between projects:

```
Assets/__IP ARCHITECTURE/
  Scripts/     all runtime code, grouped by subsystem
  Editor/      custom inspectors and editor windows for the framework
  Prefabs/     framework prefabs (StatDisplayerFull, TextAlert, ...)
  Materials/   master shader, outlines, dissolve, VFX, shader subgraphs
  Sprites/
```

The top-level `Assets/_Scripts/`, `_Prefabs/`, `_Sprites/`, `_Audio/` folders are reserved for **project-specific** content and are currently empty. Put game-specific code in `Assets/_Scripts/`; put reusable systems in `Assets/__IP ARCHITECTURE/Scripts/<subsystem>/`.

Note: all game scripts compile into a **single default assembly** — there are no `.asmdef` files outside third-party packages (VolFx, GlitchFx, PrimeTween). Every script sees every other script and all class names share one global namespace.

### Spatial convention (important)

The world is **3D with a perspective camera** and billboarded sprites: characters move on the **XZ plane** (X = right/left, Z = forward/back), **Y is vertical (up)** and is normally held constant during movement.

- Build offsets/directions in XZ and keep Y fixed (e.g. `Random.insideUnitCircle` returns a `Vector2` → map it to `new Vector3(v.x, 0f, v.y)`, not `(v.x, v.y, 0)`).
- Sprites face the camera via `FacedAlwaysToCamera`; horizontal facing is a **Y-axis rotation** applied to an inserted pivot by `FlipByVelocity3D` (not a sprite `flipX`).
- Prefer PrimeTween's `Tween.Position` for moving transforms (see `MoveWithTarget`).

**Caveat:** a few files still use **2D physics** (`Rigidbody2D` / `Collider2D`) left over from the framework's 2D origin — `Unit`, `SpawnerManager` (`CircleCollider2D` map bounds), `CountingArea`, `SmartCollider`, `Pickable`, `InteractAreaTrigger`, parts of `h`. When touching those, check which physics dimension the object actually uses before adding colliders or raycasts. New code should prefer 3D physics.

## Building / Running / Testing

There is no CLI build script — this is a standard Unity project. Open it in the Unity Editor (version `6000.2.12f1`, pinned in `ProjectSettings/ProjectVersion.txt`) and press Play, or use **File > Build Settings** to build.

- **Tests** use the Unity Test Framework (`com.unity.test-framework`). Run them from the Editor via **Window > General > Test Runner** (EditMode / PlayMode tabs). There are no test assemblies in the game code, so tests would run in the default assembly.
- Headless: `Unity.exe -runTests -projectPath . -testPlatform EditMode -testResults results.xml -batchmode`.

## Global static helpers (used everywhere)

- **`h`** (`Scripts/Helpers/Helpers.cs`, ~1600 lines) — the project's utility class. Logging (`h.Out(...)` — use instead of `Debug.Log`), math (`Sign/Min/Max/Sum/Range`), randomness (`RandChoice`, `RandChance`, `RandomChoiceWithDistribution`, `RandomPositionInCircle`), hierarchy search (`FindChildWithTag`, `FindClosestByTag`, `OrderByDistance`), component/SO helpers (`AssignComponent`, `CopySO`), tweened transforms (`SmoothScaling`, `SmoothTranslating`, `FadeIn/FadeOut`), and **`h.CreateStaticInstance(this, ref Instance)`** — the singleton pattern used by every manager.
- **`P`** (`Scripts/Helpers/Preferences.cs`) — static "Preferences": global enums and lookup tables. `P.Team` + `TeamContainer` (ally/enemy tag sets, `P.GetTeamContainerByTeam`), `P.EquipmentType` / `P.equipmentDisplayType`, `P.Stat` + **`P.StatK`** (stat *string keys*), `P.raritiesNames` / `P.rarities` (`dusted` / `vanguard` / `x-grade`), `P.StatusEffects`, `P.MoodType`, `P.PositionType`, `P.colors`, and project tunables (`P.maxAttackers`). **Stats are referenced by string key (`P.StatK[P.Stat.health]`), never by raw literal.**
- **`R`** (`Assets/Generated/R.cs`) — **auto-generated**, do not edit by hand. A typed tree mirroring `Assets/Resources/` (`R.ARCHITECTURE.Audio.Mixers`, `.Materials`, `.Prefabs.UI`), regenerated by the editor Resource Class Generator configured in `Assets/ResourceClassGeneratorConfig.asset`.
- **`G`** (`Scripts/Static/G.cs`) — currently empty; intended as a global access point.
- **`HTML`** (`Scripts/Text&HTML/HTML.cs`) — rich-text tag constants and the custom dialogue markup (`pause`, `anim:` with `shake/wave/bounce/fade/rainbow`).

## Manager singletons

Managers create their `static Instance` in `Awake()`/`Start()` via `h.CreateStaticInstance(this, ref Instance)` (enforces uniqueness, optional `DontDestroyOnLoad`). Access as `GameFlowManager.Instance`.

`GameFlowManager`, `ScreenManager`, `CameraController`, `FogOfWarManager`, `DialogueManager`, `CutSceneManager`, `SFXManager`, `BGMManager`, `CursorDnDManager`, `TextAlertManager`.

Not singletons (plain components, may exist several times): `GlobalEventManager`, `InteractionManager`, `PurchaseManager`, `SpawnerManager`, the `*VolumeManager` family. `CameraManager` is an empty stub.

- **`GameFlowManager`** owns the `State` enum (Intro/Game/Pause/Loss/Finale). Long-running loops/coroutines must check `GameFlowManager.Instance.IsPaused()` / `IsGame()` and stall while paused (see `DirectorBase.TokenIncome`).
- **`GlobalEventManager`** bridges Input System actions to C# events (e.g. `OnInteractPressed`). Actions are defined in `Assets/InputSystem_Actions.inputactions`.
- **`ScreenManager`** fade in/out; builds its own overlay canvas if none is assigned.

## Subsystems (`Assets/__IP ARCHITECTURE/Scripts/`)

### Directors / token economy — `Directors/`

The core of procedural spawning and selection: **token-based weighted-random selection**.

- **`TokenBasedContainerBase<T>`** (`ScriptableObject`) holds `List<TokenBasedItemData<T>>`, each with `value`, `weight`, `cost`. Items auto-sort by cost and auto-assign `id`/`name` in `OnEnable`. Concrete assets: `GameObjectTokenContainer`, `ScriptableObjectTokenContainer`, `StringTokenBasedContainer` (menu **Containers/...**).
- **`DirectorBase<T>`** accrues `tokens` over time (`TokenIncome` coroutine, respects pause) and spends them via `ChoosePrefab` / `ChooseSeveralPrefabs` / `ChoosePrefabsUntilEmpty`. Selection filters by affordability and a "too cheap" threshold (`tokens / tooCheapCoef`), then picks by weight. Subclasses: `EntityDirector<T>`, `GameObjectDirector`, `StringDirector`, `UnitDirector`.
- **`EntityManager<T>` / `GameObjectManager` / `UnitManager`** track spawned counts against `maxEntityCount`, instantiate under `unitParent`, and wire `DestroyNotifier` callbacks to keep counts in sync.
- **`IConsumeAllTokens`** + `EntityEventBase` / `DefaultEvent` — one-shot events that drain a director's token pool.
- **`SpawnerManager`** — rate-based spawning inside/outside a circular map region, driven by `ObjectsWithDistributionContainer`.

### Units & stats — `Units/`, `_Scriptable objects/`, `UI/`

- **`Unit`** (`Units/UnitMonoB.cs` — the class is `Unit`, not `UnitMonoB`) is the central gameplay entity: extends `StatUIShownBase`, implements `IDamageable`. Holds `baseStats` / `stats` / `resources` as `ObjectsCountContainer`s (all `Instantiate`d per-unit in `Awake`, so runtime edits never touch the asset), a `P.Team` + `TeamContainer` with extra enemy tags, an equipment-slot system keyed by `P.EquipmentType`, and `UnityEvent`s (`onTakeDamage`, `onHeal`, `onUpdStats`, `onChangeStat`, `onSetStat`, `onDeath`).
  - Read stats through `GetStat` / `GetBaseStat`, write through `ChangeStat` / `SetStat` (these fire the events and set `lastChangedStatKey`).
  - Death: `Die()` runs every factory in `coroutinesBeforeDeath` plus the `Death`-typed `AnimationControllerBase` concurrently and waits for all of them before finalizing. Register death animations there rather than hooking `onDeath` teardown.
- **`ObjectsCountContainer`** (`ScriptableObject`) — the string→float dictionary behind all stats. Built from parallel `rawKeys`/`rawValues` lists into `values`; can initialize from an `ObjectsKindsContainer`, a `KeyContainer`, or another container, with optional randomized distribution (`randValues`, `minSum`/`maxSum`, `maxValue`).
- Sibling containers: `ObjectsKindsContainer`, `ObjectsWithDiscributionContainer`, `ScriptableObjectsKindsContainer`, `KeyContainer`, `StoryContainer` (menu **Scriptable Objects/...**).
- **`StatUIShownBase`** — the contract for anything displayed in stat UI (`GetStats`, `GetSprite`, `GetMaterial`, `GetName`); consumed by `StatUIDisplayer`, `UnitStatBar`, `Bar`, `BarTracker`.

### Equipment & status effects — `Equipment/`

`EquipmentControllerBase` (`ScriptableObject`, menu **Equipment/Equipment Controller**) is the unit of inventory: it owns `EquipmentBase` prefabs, an icon/material, a `P.EquipmentType`, and its DnD object prefab. `Init` creates instances, `Apply`/`Remove` attach to a `Unit`. Effects derive from `StatusEffectBase` (e.g. `UnitStatChangeStatusEffect`). `MultipleEquipmentInTail` handles multi-instance placement.

### Drag & drop — `DnD/`

`CursorDnDManager` (singleton) tracks the dragged/hovered object and click state; `DragableObject` + `SlotBase` are the base pair, specialized by `EquipmentDNDObject`/`EquipmentDNDSlot` and `GameObjectDNDObject`. `IRecyclable.OnRecycle` handles scrapping.

### Purchasing — `PurchaseItems/`

`PurchaseManager` rolls rarities through a `StringDirector` against a token budget, then picks a concrete item per (type, rarity) out of a nested `ScriptableObjectsKindsContainer`. `PurchaseItemBase` is the item asset; `PurchaseItemUIWIndow`, `PricingListViewer`/`PricingListItem` render the shop.

### Dialogues & cutscenes — `Dialogues/`, `CutScenesScripts/`

`DialogueContainer` / `DialogueContainerWithModels` (menu **Dialogues/...**) hold `DialogueNode` sequences authored in the custom **`DialogueEditorWindow`**. `DialogueManager` (singleton) types text out (`TextAnimation`, `HTML` tags, per-speaker voice clips), gates input during node-enter transitions, and drives `Talkable` / `TalkableWithModels` participants. `CutSceneManager` runs `CutSceneBase` assets by reference or by `Resources` path (`Cutscenes/...`).

### Animation — `Animation/`

`AnimationBase` (PrimeTween-driven, with initial-state capture and `returnToTheInittialState` behaviour) + `AnimationControllerBase`, which collects child `AnimationBase` components filtered by `AnimationPreferences.Type` and plays them as a group. Concrete animations: `Scaling`, `Squish`, `Shake` (+ UI), `Rotating`, `Floating`, `Dissolve` (+ UI), `SpriteChange`, `UIImageSpriteChange`, `CirclingMovement`, `OnHitAnimation`, `AttackAnimationController`, `FlipByVelocity3D`.

### Rendering & materials — `Rendering/`, `Materials/`

`VolumeManager` is a **per-Volume component** (not a singleton — a scene has several volumes) that reads/sets/tweens URP override values through the runtime profile copy, so the profile asset on disk is never modified. Typed subclasses: `BloomVolumeManager`, `VignetteVolumeManager`, `GlitchVolumeManager`, `JpegVolumeManager`, `VhsVolumeManager`.
`MasterMaterialController` drives the MasterMat shader (brightness, contrast, dissolve, ...) across both `Renderer`s and UI `Graphic`s — UI needs the `MasterMatUI` variant for stencil/mask support. Also `OutLineController`, `EatingMatController`, `TMProMaterialController`.

### Camera — `Camera/`

`CameraController` (singleton) owns zoom (clamped, PrimeTween-eased) and target assignment; `CameraFlowTargeting` does the smoothed follow; `BalatroScreenShake` and EZ Camera Shake provide shake; `FacedAlwaysToCamera` billboards sprites with per-axis locking. Cinemachine is installed but **not currently used**.

### Fog of war — `FogOfWarExtended/`

Thin wrapper over the third-party **AOSFogWar** (`FischlWorks_FogWar.csFogWar`): `FogOfWarManager` (singleton) adds/removes `csFogWar.FogRevealer`s, reading sight range from a `FogOfWarRevealer` component when present.

### Audio — `Audio/`

`AudioManagerBase` (source creation + `PlayClipIndependently`) is subclassed by `SFXManager` and `BGMManager` (crossfading A/B sources). Mixer groups come from `AudioMixerManager` (`MainMixer` in `Resources/ARCHITECTURE/Audio/Mixers`). `AudioClipController` / `AudioResource` wrap clip selection.

### Interaction — `Interaction/`

`IInteractable` implementers plus `InteractableObj`, `InteractAreaTrigger`, and `InteractionManager`, which pools `InteractionIcon` prompts.

### Text alerts, UI, misc helpers

`TextAlertManager.CreateDamageAlert` spawns floating `AlertDamageText`. UI: `Bar`, `UnitStatBar`, `StatUIDisplayer`, `SmartButton`, `SliderDragAnimation`, `ScalingText`, `DisableRaycast`. Helpers: `DestroyNotifier`, `MoveWithTarget`, `PosTracker`, `Pickable`, `SmartCollider`, `AreaBase`/`CountingArea`, `CustomEvents`, `Wrappers`, `SetDontDestroyOnLoad`, `RotatingGO`, `ScaleMult`.

### Interfaces — `Helpers/Interfaces.cs`

The behaviour contracts, and the first place to look when wiring a new component: `IDamageable` (`MayDamage`, `TakeDamage`), `IInteractable` (`OnInteract`, `Start/End/ContinuousInteraction`), `IEquipment` (`unit`, `Apply`, `Remove`), `IOnMove`, `IOnLand`, `IRecyclable`, `IBeforeDestroy`.

### Editor tooling — `Assets/__IP ARCHITECTURE/Editor/` and `Scripts/Dialogues/Editor/`

Custom inspectors for `ObjectsCountContainer`, `TokenBasedContainer`, `EntityDirector`, `GameObjectDirector`, `PurchaseItemBase`, `DialogueContainer`, `StoryContainer`; plus `DialogueEditorWindow` and `UnitPrefabGeneratorWindow` (its `MenuItem` is currently commented out).

## Third-party packages (context that affects code)

In `Assets/`:

- **PrimeTween** (`Plugins/PrimeTween`) — tweening library used throughout animation, camera and UI code. Prefer it over coroutines for new tween-style animation.
- **AOSFogWar** (`AOSFogWar`, namespace `FischlWorks_FogWar`) — grid fog of war; wrapped by `FogOfWarExtended/`.
- **VolFx / ScreenFx / Tools** and **GlitchFx** (`VolFx/`, `Glitch/`) — URP post-processing passes and volume components; these DO have their own asmdefs.
- **CRT-Free** (`CRT-Free/`) — CRT screen effect.
- **EZ Camera Shake** (`EZ-Camera-Shake-Unity-master`) — camera shake; already imported inside `h`.
- **SpriteOutline** (`SpriteOutline/Shaders`) — sprite outline shader used by `OutLineController`.

Package manager: URP 17.2.0, Input System 1.14.2, Cinemachine 3.1.7 (unused), AI Navigation 2.0.9 (unused — no NavMesh code yet), Timeline, Visual Scripting, TextMeshPro/uGUI, Test Framework, and **rsg** (`com.xk.rsg`, git dependency — reactive/promise utilities).

## Conventions

- Use **`h.Out(...)`** for logging, not `Debug.Log`.
- Reference stats / teams / equipment types / rarities through **`P`**, never with magic strings.
- Never hand-edit `Assets/Generated/R.cs`; regenerate it instead.
- New managers use the `h.CreateStaticInstance` singleton pattern to match existing ones.
- Reusable systems go under `Assets/__IP ARCHITECTURE/Scripts/<subsystem>/`; project-specific gameplay code goes under `Assets/_Scripts/`. Grouping folders keep the `_` prefix (`_Prefabs`, `_Sprites`, `_Audio`).
- `ScriptableObject` data that a runtime object mutates must be `Instantiate`d first (see `Unit.Awake`) so the asset on disk is never modified.
- Long-running coroutines check `GameFlowManager.Instance.IsPaused()` and idle while paused.

# TASK Comment Processing Rules

When working in this codebase, treat every comment that begins with `//TASK` as an explicit instruction for you.

Requirements:

1. Scan all files you read for comments matching:

   * `//TASK`
   * `// TASK`

2. Collect every `//TASK` comment as an actionable task.

3. Before making changes:

   * Create a complete checklist of all discovered `//TASK` items.
   * Review the surrounding code to understand the intent of each task.

4. While implementing:

   * Complete every `//TASK` item unless it is impossible, contradictory, or lacks required information.
   * Do not ignore any `//TASK` comment, even if it is not near the code you are currently editing.

5. After implementation:

   * Verify each `//TASK` has been addressed.
   * Remove the `//TASK` comment if the task is fully completed, unless instructed otherwise.
   * If a task cannot be completed, explain why and describe the remaining work.

6. During status updates:

   * Report discovered `//TASK` items.
   * Indicate which are completed, in progress, or blocked.

7. Consider `//TASK` comments higher priority than ordinary comments, TODOs, or suggestions.

Your workflow should always be:

Scan files → Collect all `//TASK` items → Create checklist → Implement → Verify completion → Report results.
